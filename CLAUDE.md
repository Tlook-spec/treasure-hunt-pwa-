# 寻宝游戏 PWA 项目说明

> 这个文件是给 Claude Code 看的「项目说明书」。每次启动新对话时 Claude Code 会自动读取这个文件，了解项目背景和约定。**不要删除这个文件**。
>
> **当前版本：v1.3**（详见文末更新记录）

---

## 1. 项目背景

这是一个**双站点的儿童户外解谜游戏项目**：

- **编辑端（admin）**：普通网页，家长在电脑/iPad/手机浏览器里设计探险和题库
- **游戏端（play）**：iPad 上的离线 PWA，孩子户外扫码答题

**核心理念**：编辑和游戏完全分离的两个独立网站，简化复杂度、避免误操作。

**目标用户**：
- **家长**（编辑者）：自己孩子的爸妈，4-12 岁孩子家长
- **孩子**（玩家）：4-12 岁儿童，**MVP 单人**，V1 多人

**数据流**：
- 编辑端浏览器 IndexedDB → 导出 JSON → AirDrop/邮件传到 iPad → 游戏端 PWA 导入
- **JSON 文件就是数据库**，浏览器只是工作台

---

## 2. 三层数据结构（v1.3 新增）

```
L1 探险（Level）              一次完整的户外活动
  ├── L2 点位（Point）        一个物理打卡点，1 个二维码 + 多道题
  │     ├── L3 题目 1（Question）
  │     ├── L3 题目 2
  │     └── L3 题目 N
  └── L2 × 3-5
```

**重要业务规则**：

- 一个 L2 包含**多道**题目（家长配置）
- 孩子扫码后**依次答完一个 L2 的所有题**才能进下一站
- 每个 L2 通关时**拍一张合影**（可选）
- 整个 L1 通关后看总分 + 用时 + 所有 L2 合影

---

## 3. 技术栈（已锁定，请遵守）

**前端**：纯 HTML + CSS + JavaScript，**不使用任何前端框架**

**核心依赖库**（**全部本地化，不用 CDN**）：

| 用途 | 库 |
|------|---|
| 本地存储 | Dexie.js（IndexedDB 封装） |
| 扫码 | html5-qrcode |
| 二维码生成 | qrcode.js |
| 图片处理 | 浏览器原生 Canvas |
| 朗读（V1） | Web Speech API |
| CSV 解析 | PapaParse |
| 离线（仅游戏端） | Service Worker |
| 调试 | Eruda（开发环境，?debug=1） |

**编辑端**：普通网页，**不做 PWA**，浏览器直接打开使用
**游戏端**：完整 PWA + Service Worker，iPad Safari 安装到主屏幕

**禁止使用**：React/Vue/Svelte、TypeScript、jsPDF（用 window.print 代替）、HMAC、UUID v4（用短 ID）

**部署**：GitHub Pages，两个端在同一仓库不同子目录

## 5. 开发者画像与协作约定

**开发者**：完全零基础，每天 1-2 小时，每周 5-8 小时，Windows 11 + VS Code + Claude Code

**协作原则**：

### 5.1 代码风格

- 写最简单的代码（不用复杂设计模式）
- 多写中文注释
- 变量名清晰（宁可长不要短缩）
- 函数 20-30 行内
- 一个函数只做一件事

### 5.2 沟通风格

- 打比方解释，不堆术语
- 报错时给操作指引
- 每次回答控制长度
- 多种方案先列选项让用户选
- Windows 环境用 PowerShell 语法

### 5.3 范围控制

- 一次只做一件事
- 不主动重构
- 加新功能不破坏旧功能
- 大改动先讨论

### 5.4 错误处理

- 先收集完整错误信息
- 不确定时直说，别瞎编
- 修复后清楚说明改了什么

## 6. 核心数据模型（简要）

详见 `docs/PRD.md` §6，这里只列概念：

- **Level (L1)**：探险，包含 password、points 数组
- **Point (L2)**：点位，包含 questionIds 数组（多题）、code（6 位数字）
- **Question (L3)**：题目，含 usedCount 数字（v1.3 简化）
- **Player**：玩家（V1 多人）
- **GameSession**：游戏会话，含 pointRecords（每 L2 内多 questionResults）
- **PhotoBlob**：照片存储（仅头像 + L2 通关合影两类）

**关键架构原则（v1.3）**：

- 短 ID（不用 UUID v4，自用零冲突）
- 每 L2 一张通关合影（不是每题都拍）
- usedCount 简单数字（不维护引用列表）
- **不做 snapshot 机制**——session 直接引用当前 L1 数据
- **导入数据保护**：导入前检查未完成游戏并二次确认

## 7. 重要业务规则

1. **顺序解锁**：按 L2 顺序扫码
2. **二维码内容**：纯 6 位数字短码（同时用于扫码 + 手动输入）
3. **6 位数字码 fallback**：扫不出时手动输入
4. **「找不到？」按钮**：输 L1 密码可跳过当前 L2
5. **求助机制**：3 次/局，标记 `usedHelp=true`
6. **答错容错**：每道题最多 3 次，第 3 次错保底通过（0 分）
7. **计分规则**：
   - 一次答对 = 10 分
   - 错 1 次后答对 = 5 分
   - **曾用求助** = 3 分（覆盖所有用过求助的情况）
   - 保底 = 0 分
8. **L2 多题**：一个 L2 含多道题，依次答完才能进下一站
9. **拍照**：每 L2 通关一张合影，可选不阻塞
10. **多人答题归属**（V1）：每道题答对后指定答对者
11. **连续答对计数**（V1）：「独占被选」时才 +1
12. **颁奖流程**（V1）：先全员投票 → 一起颁奖
13. **保底奖**（V1）：无奖玩家自动获保底
14. **题库 usedCount**：题目被绑定到 L2 时 +1，移除时不减
15. **删除 + Undo**：删除前确认 + 5 秒撤销 toast
16. **导入数据保护**：游戏端导入 JSON 前检查未完成游戏

```

1. 提示用调试网址打开：网址后加 ?debug=1

3. 让用户截图 Console 错误

```

## 11. 禁止事项

- ❌ React / Vue / Svelte 等框架
- ❌ 大型 UI 库（Bootstrap、Tailwind 完整版、Material UI）
- ❌ TypeScript
- ❌ 付费服务
- ❌ 复杂代码炫技
- ❌ 一次性写 500+ 行新代码
- ❌ 删除或重命名用户文档
- ❌ 主动修改非当前任务范围的代码
- ❌ MVP 实现 V1 / V2 功能
- ❌ 使用 CDN 引入第三方库
- ❌ 使用 jsPDF（用 window.print）
- ❌ 使用 HMAC（用 6 位数字短码 + 数据库查询）
- ❌ 使用 UUID v4（用短 ID）
- ❌ 全量 snapshot 机制（直接引用即可）
- ❌ 编辑端做 PWA / 游戏端做编辑功能（双站点必须分离）
- ❌ 在 admin/ 目录写游戏代码或 play/ 目录写编辑代码（除非是 shared/ 共用）
