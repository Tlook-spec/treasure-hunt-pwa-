# 双分支发布说明（release = 孩子玩 / main = 开发）

> 一句话：**main 是你的开发版,release 是孩子玩的发布版,两个分支各自部署到不同网址,互不影响。**

---

## 1. 两个分支、两个网址

| 分支 | 部署到 | 网址 | 谁用 | 名字/图标 |
|------|--------|------|------|-----------|
| `release` | 网站**根目录** | `https://tlook-spec.github.io/treasure-hunt-pwa-/play/` | 🧒 孩子玩 | 正常「寻宝游戏」+ 蓝色图标 |
| `main` | 根目录下 **`/dev/`** | `https://tlook-spec.github.io/treasure-hunt-pwa-/dev/play/` | 🛠️ 你开发测试 | 「寻宝-DEV」+ 橙色图标 |

编辑端同理：`.../treasure-hunt-pwa-/admin/`（release）、`.../treasure-hunt-pwa-/dev/admin/`（main）。

部署由 GitHub Actions 自动完成（`.github/workflows/deploy-pages.yml`）：
**每次推送 main 或 release，都会重新组装整站**——release 内容放根目录、main 内容放 `/dev`。

---

## 2. DEV 名字/图标是怎么来的（重要原理）

DEV 的名字（寻宝-DEV）和橙色图标**不在任何分支的源码里**，是工作流**部署时只往 `/dev` 副本注入**的
（`manifest.json` 和 `index.html` 用 `sed` 改名字 + 换图标路径，指向 `icon-dev-*.svg`）。

**为什么这么做**：这样将来你用 `git reset --hard main` 把开发版全量同步给孩子时，
**绝不会**把 DEV 标记带到 release——两个分支的 `manifest.json` / `index.html` 永远是正常版。
你完全不用手动维护两套 manifest。

## 3. 平时两种操作（直接照抄命令）

> 都在 VS Code 终端（PowerShell）里跑。平时你在 `main` 上开发。
> ⚠️ 不管哪种，**要让孩子真正看到新版，必须像平时一样改 `play/service-worker.js` 的 `CACHE_VERSION`**（小数 +0.01）。

> 🩸 **血泪教训（2026-07-19 踩过一次，务必记住）：改完文件，第一件事永远是先 `git add .` + `git commit`，不能直接 `git push`。**
> - 为什么：`git push` 只推送「**已提交（commit）的历史**」。你只改了文件、没 commit，push 会显示 `Everything up-to-date`（以为推成功了，其实啥也没推）。
> - 更危险的是接下来那句 `git reset --hard main`——它会**把所有「没提交的修改」直接抹掉**。曾经就这样丢过一整轮改动（靠 VS Code 本地历史才找回）。
> - **自检口诀**：改完先跑 `git status`，看到你改的文件是绿色/在「Changes to be committed」= 已 add；`git commit` 之后再跑一次 `git status`，显示 `nothing to commit, working tree clean` = 真的进历史了，这时 `push` 才有意义。
> - 拿不准就一句话让 Claude 帮你跑：「帮我提交并推送，commit 写 xxx」。

### (a) main 修了个 bug，只想把这一个修复给孩子（不带 C/D 半成品）

```powershell
# 1. 在 main 上修好 bug，改 CACHE_VERSION，提交、推送
git add .
git commit -m "fix: 描述这个修复"
git push

# 2. 查出刚才这个修复 commit 的编号（最上面一条，复制前 7 位）
git log --oneline -5

# 3. 切到 release，把这一个 commit「摘」过去
git checkout release
git pull
git cherry-pick 粘贴commit编号

# 4. 推 release（触发部署到孩子的网址）
git push

# 5. 切回 main 继续开发
git checkout main
```

`cherry-pick` = 只摘这一个修复，不会把 C/D 没做完的代码带给孩子。

### (b) C/D 全做好了，把 main 全部内容更新给孩子

```powershell
# 1. 确保 main 已推送、CACHE_VERSION 已改
git checkout main
git push

# 2. 切到 release，让它和 main 变成完全一样
git checkout release
git pull
git reset --hard main
git push --force-with-lease

# 3. 切回 main
git checkout main
```

`reset --hard main` 让 release 直接等于 main。因为 release 只是「发布镜像」、没别人往里提交，
这样最简单不会乱。`--force-with-lease` 是带安全锁的强制推送。

> ⚠️ **跑第 2 步的 `reset --hard` 之前，先确认 main 上要发的东西都已经 `commit` 了**（`git status` 显示 clean）。
> 因为这个命令会连带丢弃当前工作区里「没提交的修改」——如果你的新改动还没 commit，会被一起抹掉。

> 拿不准时随时让 Claude 帮你跑这些命令，说「帮我把这个 bug 同步到 release」或「把 main 全量发布给孩子」即可。

## 4. PWA / Service Worker：孩子那边会怎样

孩子的 App 装在根目录 `/play/`，开发版在 `/dev/play/`，两者缓存范围完全独立。

- **你更新 main（开发版）→ 孩子完全不受影响。** 只改了 `/dev`，孩子的 App 根本不会去看，SW 不更新、缓存不变。
- **你更新 release → 孩子要这样才拿到新版：**
  1. 你必须改了 `CACHE_VERSION`（否则 SW 认为没变，孩子还是旧缓存）。
  2. 孩子 iPad **在有网情况下**打开 App → SW 后台发现新版 → 下载新缓存。
  3. **把 App 从后台彻底划掉再重开一次** → 生效。
  4. 去设置页看 `BUILD_TAG` 变成新的 = 更新成功。
  - ⚠️ 只在飞行模式开 App 永远收不到更新，必须联网开一次。

## 5. 首次配置 / 换设备：iPad 主屏图标

- 在 `.../treasure-hunt-pwa-/play/` 用 Safari「添加到主屏幕」= 孩子的正式版（蓝图标「寻宝游戏」）。
- 在 `.../treasure-hunt-pwa-/dev/play/` 用 Safari「添加到主屏幕」= 开发版（「寻宝-DEV」）。
- ⚠️ **如果主屏上已有旧图标**：iOS 会缓存旧的名字/图标，可能要**先删掉旧图标，再重新「添加到主屏幕」**才会显示新名字/图标。
- 注：DEV 图标是 SVG，iOS 对 SVG 主屏图标支持不稳定；万一 DEV 图标没变成橙色，**至少名字会显示「寻宝-DEV」**，仍能区分。要 100% 保证图标也不同，需改用 PNG 图标（以后可补）。

## 6. GitHub 网页一次性配置（只需做一次，已于 2026-06-22 完成 ✅）

这套部署已经配好并验证通过。下面记录当初点了哪些开关，**换设备/重建仓库时照做**：

**① 把 Pages 来源切到 GitHub Actions**
仓库 → **Settings** → 左侧 **Pages** → **Build and deployment → Source** → 选 **GitHub Actions**。

**② 放开 github-pages 环境的分支限制（关键！否则 release 部署会被拒）**
仓库 → **Settings** → 左侧 **Environments** → 点 **`github-pages`** →
**Deployment branches and tags** 一段 → 允许 **`main` 和 `release` 两个分支**
（要么选 "No restriction"，要么用 "Selected branches and tags" 把 `main` 和 `release` 都加进去）。

> 当初只配了 ① 没配 ②，`release` 分支部署报错
> `Branch "release" is not allowed to deploy to github-pages due to environment protection rules`。
> 加上 ② 后重跑（Actions → 红色记录 → Re-run all jobs）即变绿。

**验证**：**Actions** 标签页每次推送都能看到部署记录，绿 ✓ 即成功；点开有部署网址。
若重跑仍报权限错：**Settings → Actions → General → Workflow permissions** 选 **Read and write permissions** 保存再重跑。
