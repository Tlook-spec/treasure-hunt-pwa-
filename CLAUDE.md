# 寻宝游戏 PWA 项目说明

> 这个文件是给 Claude Code 看的"项目说明书"。每次启动新对话时 Claude Code 会自动读取这个文件，了解项目背景和约定。**不要删除这个文件**。

> **当前版本：v1.1**（详见文末更新记录）

## 1. 项目背景

这是一个 iPad 上**离线运行**的儿童户外解谜游戏 PWA（渐进式 Web 应用）。

**使用场景**：家长提前在 App 里出题、生成二维码、打印贴到户外打卡点（小区、公园等）。孩子拿着 iPad 在户外按顺序找点、扫码、答题，全程不需要联网。

**目标用户**：
- **家长**（编辑者）：自己孩子的爸妈，4-12 岁孩子的家长
- **孩子**（玩家）：4-12 岁儿童，可能单人玩，也可能 2-6 人组队玩

**核心价值**：把户外探索、亲子互动、学科学习融合成一种孩子主动想玩的游戏。

**双端使用**（v1.1 新增）：
- **桌面浏览器端**（家长用）：高效的关卡设计、题库批量管理、CSV 导入、PDF 打印
- **iPad 端**（孩子用）：游戏体验、扫码、答题、拍照、通关庆典
- 两端是**同一个 PWA**，通过 JSON 导入导出实现数据同步（MVP 方案）

## 2. 技术栈（已锁定，请遵守）

**前端技术**：纯 HTML + CSS + JavaScript，**不使用任何前端框架**（不要引入 React、Vue、Svelte 等）

**核心依赖库**（按需引入，能不用就不用）：

| 用途 | 推荐库 | 备注 |
|------|--------|------|
| 本地存储 | Dexie.js | IndexedDB 的简易封装 |
| 扫码 | html5-qrcode | 浏览器扫二维码 |
| 二维码生成 | qrcode.js | 生成二维码图片 |
| PDF 导出 | jsPDF | 导出打印用 PDF |
| 图片处理 | 浏览器原生 Canvas API | 加水印、压缩 |
| 文字朗读 | Web Speech API | 浏览器原生 TTS |
| 离线支持 | Service Worker | 浏览器原生 PWA 能力 |
| CSV 解析 | PapaParse（CDN）| 题库 CSV 批量导入 |
| 调试工具 | Eruda（开发用）| iPad 上嵌入式调试控制台 |

**构建工具**：暂不使用任何构建工具（不用 Vite、Webpack 等），保持纯静态文件，浏览器直接打开就能用。

**部署平台**：GitHub Pages（免费 HTTPS 静态托管）

**目标设备**（v1.1 更新）：
- **桌面浏览器**（家长管理）：Chrome / Edge / Safari，1280px+ 宽屏布局
- **iPad**（孩子游戏 + 偶尔家长微调）：Safari 浏览器，竖屏优先
- **iPhone / Android 平板**（备用）：响应式适配

**朝向**：
- **游戏模式**：**竖屏优先**，横屏作为"也能用"。所有页面必须在竖屏下体验最优，横屏下不能错乱。
- **家长模式**：**桌面横屏布局优先**，iPad 上响应式适配。

## 3. 开发者画像与协作约定

**开发者背景**：
- 完全零基础，无任何编程经验
- 无 App 开发经验、无 UI 设计经验
- 之前用过 Claude Code 做过简单网页游戏
- 工作之余开发，每天投入 1-2 小时，每周 5-8 小时
- 开发环境：Windows 11 + VS Code + Claude Code 扩展 + Claude Code CLI

**核心协作原则**（请严格遵守）：

### 3.1 代码风格

- **写最简单的代码**：不要使用复杂的设计模式、闭包技巧、链式调用
- **多写中文注释**：每个函数前用注释说明它干什么；复杂逻辑逐行注释
- **变量名用易懂的命名**：宁可长一点也要清晰，如 `currentPointIndex` 不要 `cpi`
- **避免炫技**：宁可代码长一点，也要让零基础能看懂
- **一个函数只做一件事**：函数行数控制在 20-30 行内

### 3.2 沟通风格

- **解释代码时打比方**：不要堆专业术语。例：解释 IndexedDB 不要说"客户端键值存储"，说"浏览器自带的小型数据库，像一个能存东西的箱子"
- **报错时给出操作指引**：不要只说"找不到模块"，要说"请运行 xxx 命令安装"
- **每次回答控制长度**：一次不要展开太多内容，等用户消化完再继续
- **主动询问**：方案有多种可能时，先列出选项让用户选，不要擅自决定
- **Windows 环境**：涉及命令时使用 PowerShell 语法，避免 Mac/Linux 的 bash 命令

### 3.3 工作范围控制

- **一次只做一件事**：用户让你做 A，不要顺手做 B
- **不要主动重构**：除非用户明确要求，不要改动已有代码的结构
- **加新功能不破坏旧功能**：每次修改后让用户验证旧功能还能用
- **大改动前先讨论**：超过 100 行代码的改动前，先讲清楚思路再写

### 3.4 错误处理

- **用户报错时**：先问"你看到的完整错误信息是什么？操作步骤是？"再修
- **不确定时**：直接说"我不确定，让我们试一下 XXX 看看"，不要瞎编
- **修复后**：清楚说明"我改了什么、为什么这样改、用户需要做什么验证"

## 5. 核心数据模型（简要）

详细字段定义见 `docs/PRD.md` §6，这里只列概念：

- **Level（关卡）**：一次完整游戏的容器，包含多个点位
- **Point（点位）**：物理空间中的一个打卡点，绑定二维码、提示文本、题目池
- **Question（题目）**：独立题库中的一道题，**含使用记录**（被哪些关卡引用过）
- **Player（玩家）**：参与某次游戏的一个孩子，含头像、名字、颜色
- **GameSession（游戏会话）**：一次完整游戏过程的所有状态，含玩家列表、当前进度、积分记录、拍照

**关键关系**：
- 一个 Level 有多个有序 Point
- 一个 Point 引用题库中的若干 Question（运行时随机抽一道）
- 一个 GameSession 关联一个 Level + 多个 Player

## 6. 重要业务规则（容易忽略，请记住）

1. **顺序解锁**：孩子必须按 Point 顺序扫码，扫到非当前点的码要友好提示"这不是你现在要找的点"
2. **二维码内容**：包含关卡 ID + 点位 ID + 校验码（防止打印假码作弊）
3. **求助机制**：每次游戏 3 次求助机会，由家长线下解答
4. **答错容错**：答错之后可以再次回答，最多 3 次回答机会，第 3 次错保底通过（0 分）
5. **多人答题归属**：每道题答对后，由孩子们指定"是谁答对的"（多选/大家一起）
6. **连续答对提醒**：同一玩家连续 3 题答对，弹友好提示鼓励让其他孩子试
7. **颁奖流程**（v1.1 调整）：通关后先全员投票 → 投票完成后一起公布所有奖项（答题之星 + 投票奖项 + 保底奖一起揭晓），增强仪式感和悬念
8. **投票规则**（v1.1 明确）：
   - 答题第一名不参与投票（已获得答题之星）
   - 每个奖项每人一票，可投自己
   - **同一个孩子可以被投多个奖项**（采用"宽松投票"规则）
   - 平票则共同获奖
9. **保底奖**：每个孩子至少有一个奖。投票结束后系统自动检查，给"一个奖都没拿到的玩家"颁特别奖（未来之星 / 最强辅助奖等），自然融入颁奖环节
10. **题目使用记录**：每道题记录被哪些关卡引用过，避免家长在不同关卡里反复用同一题导致孩子背答案
11. **数据导出备份**：因 iOS PWA 数据可能被系统清理，必须支持关卡和题库的 JSON 导出导入；MVP 阶段为手动导出导入

## 8. 视觉设计规范

> 详细规范待用户在阶段 1 完成线框图后补充。当前采用以下临时规范：

- **主色**：待定（用户将在阶段 1 选定）
- **字体**：系统默认无衬线字体（Windows 微软雅黑 / iOS 苹方）
- **风格**：友好、明亮、儿童友好，避免深色调
- **圆角**：12-16px（适度圆润）
- **按钮最小尺寸**：60px × 44px（户外大手指操作）
- **重要文字最小字号**：16px（户外强光下可读）

**响应式断点**：
- 桌面（家长用）：≥1280px
- 平板横屏：768-1279px
- iPad 竖屏：481-767px
- 手机：≤480px

## 9. 与 Claude Code 协作的具体场景示例

**场景：用户让你创建新功能**

```
用户："请帮我加一个求助按钮"

期望响应：
1. 先确认需求："我理解你要的是：在答题页右下角加一个'求助'按钮，
   点击后求助次数 -1，弹出'快去问爸爸妈妈！'的提示，对吗？"
2. 等用户确认
3. 修改对应文件，每个改动用中文注释
4. 改完后告诉用户："已修改 XX.html 和 XX.js，请在浏览器刷新查看效果。
   你可以测试：1) 按钮位置对吗？2) 点击后次数减少了吗？3) 提示弹出了吗？"
5. 更新 PROGRESS.md
```

**场景：用户报错**

```
用户："按了求助按钮没反应"

期望响应：
1. 先收集信息："请按 F12 打开开发者工具，切到 Console 标签，
   再点一次求助按钮，看看有没有红色错误信息？截图给我看。"
2. 根据信息定位问题
3. 给出修复方案，并解释原因（用比喻方式）
```

**场景：iPad 上调试**

```
用户在 iPad 上遇到问题报错。

期望响应：
1. 提示用户用调试网址打开：在网址后加 ?debug=1
2. 引导用户打开 Eruda（点右下角齿轮）
3. 让用户截图 Console 里的红色错误
4. 根据错误诊断问题
```

## 10. 禁止事项

请**永远不要**做以下事情：

- ❌ 引入 React、Vue、Svelte 等框架（即使用户问起也要解释为什么不引入）
- ❌ 引入大型 UI 库（如 Bootstrap、Tailwind 完整版、Material UI）
- ❌ 使用 TypeScript（保持纯 JavaScript）
- ❌ 引入需要付费的服务（保持零成本）
- ❌ 把代码写得"很高级"但难维护
- ❌ 一次性写出 500+ 行新代码（必须拆分多个任务）
- ❌ 删除或重命名用户提供的文档（PRD.md、PROGRESS.md、CLAUDE.md 等）
- ❌ 主动修改不是当前任务范围的代码
- ❌ MVP 阶段实现 V2 的功能（如 GitHub 自动同步），保持简单优先
