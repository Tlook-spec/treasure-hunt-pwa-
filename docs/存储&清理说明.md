# 存储 & 清理说明

> 给「完全零基础」的你看的：游戏数据都存在哪、哪些会越攒越多、怎么清。
> 一句话先说结论：**真正占空间的是「照片」（头像 + 合影），它们存在 iPad 浏览器里、不进 JSON 文件。设置页有个「🧹 清理游玩垃圾」按钮，安全地删掉用不到的旧照片。**

---

## 1. 数据都住在哪

| 端 | 存在哪 | 存什么 |
|----|--------|--------|
| **游戏端 play（iPad）** | iPad 浏览器的 IndexedDB（库名 `treasure-hunt-play-db`）| 探险 `levels/points/questions` + 游戏存档 `sessions` + 照片 `photos`（头像 / 合影）|
| **编辑端 admin（电脑）** | 电脑浏览器的 IndexedDB（库名 `treasure-hunt-admin-db`）| 探险 `levels/points/questions`（**没有照片表**）|
| **JSON 文件** | 你 AirDrop / 邮件传的那个文件 | 只有 `levels/points/questions`（**照片不进 JSON**）|
| **GitHub 仓库** | 云端 + 电脑 git 目录 | 只有源代码（HTML/JS/CSS），**探险数据不进 git** |

要点：
- 这是**磁盘存储**，不是内存（RAM）；只有打开页面时才加载进 RAM。
- 两端物理隔离（不同库名），admin 和 play 数据不互通，只能通过 JSON 文件中转。
- **照片只在 iPad 游戏端有**（头像、通关合影）；admin 电脑端没有照片，所以电脑端不会有照片堆积。

---

## 2. 哪些数据会越攒越多

### iPad 游戏端（占空间大头 = 照片 blob）
1. **孤儿合影 / 孤儿头像**（最该清的）
   - 照片存在 `photos` 表，是**单独的行**，靠 `ownerId` 关联。
   - 整库导入会清空旧存档 `sessions`、放弃存档会删存档，但**都不删照片** → 照片变成「没人引用的孤儿」，白占空间。
   - 一张合影约 150–400 KB；玩很多次 / 反复导入，孤儿会越攒越多。
2. **历史存档 `sessions`**：每打通一局都留一条记录（无照片本身，体积小）。这是「游戏历史」，默认保留。

### 导出的 JSON 文件（变大的原因）
1. **整库导出会带上整个题库**，包括没被任何点位用到的「死题」。题库越大文件越大。
   - 想要更小更干净：用**单 L1 导出**（只带这个探险用到的题）。
2. **base64 图片**：地图底图（≤1600px，最大）+ 封面 / 提示图 / 奖图（各 ≤800px）。配得越多文件越大（都是离线必需，但心里有数）。
   - 这些 base64 是**直接存在 level/point 记录里的字段**，删掉 / 覆盖那条记录时图片跟着没，**永远不会变孤儿**（和照片不同）。

### 电脑编辑端
1. **复制 L1**：地图 / 封面 / 奖图的 base64 会**整份复制**，体积翻倍。
2. **题库里没被引用的题**：删 L1 不删题（共享题库设计），未引用的题会积累，也会跟着整库导出进 JSON。

## 3. 怎么清（三档力度）

| 档 | 怎么做 | 删什么 | 保留什么 |
|----|--------|--------|---------|
| **A 最轻·只清垃圾**（推荐）| 设置页 →「🧹 清理游玩垃圾」| 没被任何现存存档引用的**孤儿头像 + 孤儿合影** | 探险、当前未完成存档、还在用的合影（孩子的纪念都在）、历史存档记录 |
| **B 中·清空游玩历史** | 暂未做按钮，需要可加 | 整个 `sessions` + 整个 `photos` | 探险（levels/points/questions）|
| **C 核弹·整机重来** | iPad 上删掉主屏幕的 PWA 图标 | 本地全部（**含探险**）| 啥都不留，要重新导入 JSON |

### A 档「🧹 清理游玩垃圾」怎么用（已上线）
1. iPad 打开游戏端 → 右下角 ⚙️ 设置。
2. 找到「🧹 存储维护」→ 点「🧹 清理游玩垃圾」。
3. 确认后，它会扫一遍所有存档，**只删没人用的旧照片**，弹出「清理了 N 张照片，腾出约 X MB」。
4. 「当前数据」里的「📷 张照片」数字会同步减少。
- **安全保证**：它只删「孤儿」（没被任何存档引用的照片）。孩子还在用的合影、当前没玩完的存档、所有探险，**一概不动**。

### 编辑端怎么清
- 用编辑器自带的**删 L1 / 删题**功能（精准，删 L1 会连它的点位和 base64 图片一起删）。
- 或浏览器「清除站点数据」——但那会把编辑器里**所有探险**清掉，**清之前务必先导出 JSON 备份**。

---

## 4. 不用担心的（顺带让你安心）
- **Service Worker 旧缓存**：每次升级 `CACHE_VERSION`，旧版本缓存会自动删，不堆积。
- **localStorage**：就几个时间戳 / 开关，忽略不计。
- **删单个 L1 的 base64 图片**：图片是 level/point 记录里的字段，删记录就一起没，不留孤儿。
- **推 GitHub**：只上传代码，不上传探险数据，不会让占用变大。

---

## 5. 给开发者的实现备注
- 清理逻辑在 `play/pages/settings.html`（`cleanOrphanPhotos()`）：扫 `sessions` 收集被引用的 `avatarPhotoId` / `completionPhotoId`，`db.photos` 里其余即孤儿，`bulkDelete` 删除。
- 故意**不删 `sessions`**：完成 / 历史存档保留（含其合影纪念）；要做 B 档「清空历史」再单独加。
- `photos.delete` / `photos.clear` 除此清理按钮外，代码其余地方均不调用（导入只清 `levels/points/questions` + `sessions`，从不动 `photos`）。
