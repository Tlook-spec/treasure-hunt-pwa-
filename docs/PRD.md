> 本文档定义产品的完整需求。Claude Code 在实现具体功能时,请查阅本文档的对应章节。
>
> **当前版本:v1.7**(详见 §10 文档更新记录)

## 1. 产品概述

### 1.1 一句话定位

**一个轻量级的儿童户外解谜游戏,包含两个独立网站**:
- **编辑端**(家长用):普通网页,在电脑/平板/手机浏览器里设计关卡和题库
- **游戏端**(孩子用):iPad 上的离线 PWA,扫码答题、户外探险

### 1.2 使用场景

家长在电脑上设计探险(关卡)、批量录入题库、生成二维码并打印;周末把二维码贴到小区/公园打卡点;孩子拿 iPad 按顺序找点、扫码、答题、拍照纪念,全程离线。

### 1.3 核心特性

- **双站点架构**:
  - 编辑端:普通网页,**不做 PWA**,浏览器打开即用
  - 游戏端:完整 PWA,**完全离线** + 安装到主屏幕
  - 数据通过 JSON 文件手动同步(编辑端导出 → AirDrop/邮件 → iPad 导入)
- **三层结构**:
  - **L1 探险(Level)**:一次完整的户外活动,含 3-5 个 L2 点位
  - **L2 点位(Point)**:一个物理打卡点,1 个二维码 + 多道题目
  - **L3 题目(Question)**:一道独立的题,被 L2 引用
- **完全离线**:游戏端户外无网络也能玩
- **顺序解锁**:必须按家长设计的顺序通关
- **自定义内容**:家长可自由创建探险、添加题目、设计提示
- **MVP 单人优先**:MVP 单人模式跑通;V1 加多人模式
- **探险地图**(V1 新增):家长可上传手绘地图,标注 L2 坐标;游戏中每完成一个 L2 自动盖章,通关时保存完整地图作为纪念

### 1.4 设计哲学

- **MVP > 完美**:先做单人闭环,验证「孩子是否真喜欢」,再扩展
- **简单 > 全面**:自用 App,砍掉所有非必要的工程化设计
- **温和 > 严苛**:答错有容错、求助有机会、孩子不会「卡死」
- **数据可控 > 体验丝滑**:电脑端是 source of truth,iPad 是消费端
- **JSON 文件就是数据库**:编辑端的浏览器只是工作台,JSON 是真正的数据持久层
- **物理隔离**:编辑端与游戏端各自独立的 IndexedDB,不共享数据库
- **拒绝级联清理**:有引用的资源不允许删除,必须先在引用方解除,避免数据不一致
- **ID 一致性**:L1 / L2 / 题目的短 ID 在创建时生成、之后永不变,是跨设备同步判重的基础
- **手绘图只画风景**:探险地图的圆圈、名称、印章全部由 App 动态渲染叠加,手绘图本身只画风景 + 图标 + 路径,避免依赖 AI 文生图遵循复杂规则
- **CDN 优先 + Service Worker 兜底**(v1.7 新增):MVP 阶段第三方库直接用 CDN 引入(带固定版本号),Service Worker 动态缓存所有访问过的资源,实现「首次有网下载、之后完全离线」。MVP 完成后视稳定性决定是否本地化

## 2. 用户角色

### 2.1 家长(编辑者)

**画像**:4-12 岁孩子的父母

**典型任务**:
- 在电脑上批量录入题库(用 Excel 整理后 CSV 导入编辑端)
- 在编辑端设计 L1 探险、配置 L2 点位、绑定 L3 题目
- 生成二维码并打印
- 出门前实地贴二维码、打印「应急小抄」(所有 L2 数字码清单)
- 导出 JSON 数据,传到 iPad 同步
- 游戏过程中陪伴孩子、应急处理(户外丢码时家长用手机/iPad 查码、或翻应急小抄)

### 2.2 孩子(玩家)

**画像**:4-12 岁儿童

**典型任务**:
- 启动游戏端 PWA、选择探险
- 看提示找下一个点位
- 扫描二维码(或输入 6 位短码)
- 答题(一个点位多道题,连续答)
- 答完所有题后拍一张通关合影
- 通关后看成绩

- Eruda 调试(开发环境)

**⚠️ MVP 串行推进原则**:

MVP 范围对零基础开发者来说较紧凑。**严禁同时推进多条线**,建议顺序:

1. 共享层(数据模型 + IndexedDB 封装 + 工具函数)

4. 联调(JSON 同步、续玩、「找不到?」按钮)
5. 真实测试 + 修 bug

每完成一个功能块 → 真机测试 → 通过后才进入下一个。

## 4. 详细功能说明

### 4.1 编辑端(admin/)

> **编辑端是普通网页,不做 PWA**。家长在浏览器里访问 `https://你的用户名.github.io/treasure-hunt/admin/` 即可使用。
> 
> **iPad 浏览器访问编辑端也可以**(响应式适配),但 iPad PWA(游戏端)里没有编辑功能。
>
> ⚠️ **编辑端使用独立的 IndexedDB**:数据库名 `treasure-hunt-admin-db`,与游戏端的 `treasure-hunt-play-db` 物理隔离。

#### 4.1.1 首页 / 导航

- 顶部 logo + 标题「寻宝游戏 - 编辑端」
- 左侧导航栏(桌面端固定 / 移动端汉堡菜单):
  - 📝 探险(L1)管理
  - 📚 题库管理
  - 📥📤 导入导出 JSON
- 主区域:默认显示「探险管理」

#### 4.1.2 L1 探险管理

**探险列表**:
- 卡片列表显示所有探险
- 每张卡片:探险名、点位数、上次编辑时间
- 操作:新建、编辑、复制、删除

**新建 / 编辑 L1 表单**(MVP 只显示前 5 项):
- 探险名称(必填)
- 简介(选填)
- 推荐人数(1-6,V1 用,MVP 默认 1)
- 推荐年龄段(4-6 / 7-9 / 10-12)
- 主题色(预设几个)
- **以下字段 MVP 阶段在数据模型预留(默认 null/空),编辑端 UI 完全不显示;V1 才在 UI 暴露**:
  - 开场故事 `openingStory` (V1)
  - 结局故事 `endingStory` (V1)
  - 探险地图 `mapImage` (V1)
  - 地图字号 `mapFontSize` (V1)
  - 地图名称默认色 `mapNameColor` (V1)
  - 地图名称高亮色 `mapNameColorCompleted` (V1)

#### 4.1.3 L2 点位编辑

**L2 列表**(在 L1 详情页内):
- 显示该 L1 的所有 L2 点位
- 上移/下移按钮(MVP 用,V1 增加桌面端拖拽)
- 操作:新建、编辑、复制、删除点位

**L2 编辑表单**(MVP 只显示前 5 部分):
- **基本信息(仅家长可见)**:
  - 点位序号(自动)
  - 点位名称(如「小区凉亭」)
  - 家长备忘(如「贴在凉亭东侧第三根柱子上」)
  **本站扫码成功提示(孩子可见)**:文本框,支持多行(对应 `discoveryText`,可空)
- **下一点提示(孩子可见)**:文本框,支持多行
- **下一点提示(孩子可见)**:文本框,支持多行
- **L2 包含的题目**:
  - 显示已绑定的题目列表(来自题库)
  - 「➕ 从题库添加题目」按钮:弹出题库选择面板(含筛选条件)
  - 「🎲 随机抽 N 道」按钮(V1,按标签随机抽,避开已用 ≥ 2 次的)
  - 每道题旁有「移除」按钮
  - 上移/下移排序(决定答题顺序)

- **以下字段 MVP 数据模型预留,UI 不显示,V1 启用**:
  - 三段自定义文案: `questionIntroText` / `completionText`
  - 地图坐标:`mapX` / `mapY`(百分比 0-1)

#### 4.1.4 题库管理(L3)

**题库主页**:
- 顶部筛选条:学科、难度、年龄段、**使用次数**(0 次 / 1 次 / 2+ 次)(v1.7 新增筛选维度)
- 列表显示所有题目
- 每行显示:题干前 30 字、标签、难度、`usedCount`
- 使用次数显示:
  - 🔵「0 次」
  - 🟡「1 次」
  - 🟠「2+ 次」(建议替换)
- 操作:新建、编辑、删除、复制、CSV 导入、CSV 导出、JSON 导出

**v1.7 修改**:删除原 v1.6 的「批量删除使用 ≥ 2 次的题目」一键按钮。家长想清理弃用题就用筛选条件筛出 usedCount=2+ 的题,一道一道删(MVP 题量不大,完全够用)。V2 可能加「勾选 + 批量删除」功能。

**题目数据结构**:见 §6.3。

**MVP 题型**:单选、判断
**V1 题型**:多选、填空 + 题目插图
**V2 题型**:拖拽连线、图片选择

**题目「消耗式」使用模型**:

> 你的工作流:一次性导入一大批题(如 100 道),创建 L1 时从大池子里挑没用过的,用了 2 次后从池子里清理掉(因为太眼熟孩子有印象)。

- 题目绑定到 L2 时,`usedCount += 1`
- 题目从 L2 移除时,**不减**(已消耗)
- 题库默认按 `usedCount` 升序显示(未用过的优先)
- 删除受引用保护规则约束(详见 §4.1.7)

##### CSV 批量导入流程

**步骤 1:下载模板**
- 模板列:学科、难度、年龄段、题型、题干、选项A、选项B、选项C、选项D、正确答案、提示、解析
- UTF-8 with BOM 编码

**步骤 2:用 Excel/Numbers 编辑** → 保存为 CSV(UTF-8)

⚠️ **重要提示**(MVP UI 必须显著展示):
- Windows Excel 默认保存的 CSV 是 GBK 编码,**会导致中文乱码**
- 正确做法:「另存为」→ 选择「CSV UTF-8(逗号分隔)」选项
- Mac Numbers 默认就是 UTF-8,直接导出即可
- 导入页面提供 Excel 截图教学(如何选 UTF-8)

**步骤 3:上传导入**
- 使用 PapaParse 解析(按 UTF-8 处理)
- 预览前 5 行
- 确认导入
- 显示「成功 X 道,失败 Y 道(行号 + 原因)」
- 如果预览显示乱码:提示「文件不是 UTF-8 编码,请按教学重新保存」

> V1 不引入编码自动检测库(jschardet 等)。家长用过 1-2 次就知道怎么保存了。

#### 4.1.5 二维码生成与打印

**编码方式**:

每个 L2 点位生成一个**6 位随机数字码**(如 `482931`)。

- 这个码同时用于:
  - 生成二维码(QR Code 内容就是这 6 位数字)
  - 让孩子手动输入(扫不出时)
- **不需要 HMAC,不需要 levelId+pointId 拼接**
- 验证逻辑:扫到的码去 IndexedDB 查 Point 表,找到对应点位即可

**Point 创建时**:随机生成 6 位数字码,检查与已有码是否冲突,冲突则重新生成(自用场景几乎不会冲突)。

**生成流程**:
- L2 编辑页有「预览二维码」按钮
- L1 详情页有「生成全部二维码」按钮
- 单个二维码可保存 PNG

**打印流程**:
- L1 详情页有「🖨️ 打印二维码」按钮
- 触发 `window.print()` + CSS 打印样式
- 浏览器原生「打印 / 另存为 PDF」对话框
- 每页 2-4 个二维码,每个码下方:
  - 大字:第 X 站
  - 中字:点位名称
  - 大字:6 位短码(如「482931」)
  - 小字:家长备忘

**应急小抄打印**(v1.7 新增):
- L1 详情页有「📋 打印应急小抄」按钮
- 一张 A4 纸,显示:
  - L1 名称
  - 表格:序号 | 点位名称 | 6 位数字码 | 家长备忘
  - 用途说明:「出门前打印或截图存手机,户外丢码/扫不出时备查」
- 同样用 `window.print()` 实现

> ⚠️ **强烈推荐家长在电脑浏览器上完成打印**。iPad Safari 的打印对话框 CSS `@media print` 支持有限,可能导致排版混乱。如果家长一定要在 iPad 上打印,建议先「另存为 PDF」再发送到打印机,而非直接打印。

#### 4.1.6 数据导入导出

**核心架构**:编辑端浏览器的 IndexedDB 是工作台,**JSON 文件是真正的数据库**。

##### 导出

**导出选项**:
- **整库导出**(在「导入导出 JSON」菜单):所有探险 + 全部题库,一份完整备份,导出为 `exportType === "full"` 的 JSON
- **题库 CSV 导出**(在「题库管理」页):仅题库,走 CSV 通道

**MVP 不提供「单 L1 导出」** —— 推到 V1(届时与「单 L1 追加导入」配套上线)。

**MVP 不提供「仅题库 JSON」选项** —— 题库的批量进出统一走 CSV,避免功能重复。

**文件格式**:单个 JSON 文件,MVP 阶段不含图片(题目无插图、地图字段为 null)。

**JSON Envelope 格式**:

```json
{
  "schemaVersion": "1.0",
  "exportedAt": 1716800000000,
  "exportType": "full",
  "data": {
    "levels": [ /* Level 数组 */ ],
    "points": [ /* Point 数组 */ ],
    "questions": [ /* Question 数组 */ ]
  }
}
```

**v1.7 修改**:
- 删除原 `exportedBy` 字段(永远只有 "admin" 一个值,无任何代码使用)
- MVP 阶段 `schemaVersion` 固定为 `"1.0"`,V1 启用地图功能时升到 `"1.1"`
- MVP 只支持 `exportType === "full"`,V1 新增 `"single-level"`

##### 导入

**MVP 只提供「整库覆盖导入」一种模式**(备份恢复 / 跨设备同步场景):
- 仅接受 `exportType === "full"` 的 JSON 文件
- 解析 envelope,检查 schemaVersion(详见 §7.5)
- 通过后弹「这将替换当前所有数据,确定吗?」二次确认
- 游戏端额外检查未完成游戏并二次确认(详见 §7.4)
- 确认后清空 IndexedDB,写入 JSON 全部数据
- 入口:
  - 编辑端:「导入导出 JSON」菜单
  - 游戏端:「⚙️ 设置」页「📥 导入并覆盖整个数据库」按钮

**V1 新增「追加导入单 L1」模式**(同步新探险场景):
- 仅接受 `exportType === "single-level"` 的 JSON 文件
- 按 **L1 ID** 判重(不是名字):
  - L1 ID 不存在 → 直接新增(默认行为,不弹窗)
  - L1 ID 已存在 → 弹覆盖确认弹窗,内容包括:
    - L1 名称(旧 vs 新)
    - 点位数变化(如「从 3 个点位变为 5 个点位」)
    - 关联题目变化(如「共 8 道题,其中 3 道新增,2 道内容会被更新」)
    - 按钮:「取消」「替换」
- JSON 中包含的题目按 **题目 ID** 判重:
  - 不存在 → 新增
  - 存在且内容相同 → 静默跳过
  - 存在但内容不同 → 在 L1 替换确认弹窗中合并显示「共 N 道题将被更新内容」
- **题目的 `usedCount` 字段取「iPad 现有值」和「JSON 值」的较大值**(v1.7 修正,避免统计失真)
- 入口(V1):
  - 编辑端:「探险管理」菜单顶部「导入探险 JSON」按钮
  - 游戏端:「⚙️ 设置」页「导入单个 L1 探险」按钮

**两端导入的差异**:
- 游戏端导入前还要检查未完成游戏(详见 §7.4)
- 编辑端没有 GameSession 表,逻辑更简单
- 题目按 ID 判重的逻辑两端完全一致

**为什么用 ID 判重而不是名字**:
- 用户可能改名,名字不稳定
- 不同 L1 / 题目可能重名
- 短 ID 是创建时随机生成、之后永不变的(详见 §1.4 ID 一致性约定)
- ID 是跨设备同步的稳定锚点

##### 中途修改 L1 的数据保护(v1.7 新增澄清)

**场景**:家长在编辑端修改了一个 L1 的题目内容,iPad 上正在玩这个 L1 的孩子怎么办?

**MVP 的处理路径**:
1. 编辑端修改 L1 内容 → 不会自动同步,iPad 上的数据**完全没变**,孩子继续按老数据玩
2. 家长要让修改生效,必须通过「整库覆盖导入」同步到 iPad
3. 游戏端检测到「整库覆盖会清除未完成游戏进度」,弹二次确认
4. 家长选「确定」→ 老 session 被清除,从新数据开始;选「取消」→ 保持老数据玩完再说

**结论**:MVP 阶段「编辑中修改 L1」不会引发数据失效,因为唯一的同步路径是「整库覆盖」,这条路径已经强制清除了所有 session。

**V1 注意点**:V1 启用「单 L1 追加导入」后,需要重新设计这个保护机制(例如导入前检查该 L1 ID 是否有未完成 session,有则弹「这个探险正在玩中,替换会清除进度,确定吗?」)。

##### 典型工作流

**新电脑/换浏览器**:
1. 打开编辑端网页
2. 导入之前的整库 JSON 备份
3. 继续工作

**编辑 → iPad 同步(MVP 唯一路径)**:
1. 编辑端「导入导出 JSON」菜单 →「📤 整库导出」→ 下载 JSON
2. AirDrop / 邮件 / 微信 / iCloud Drive 传到 iPad
3. iPad 游戏端「⚙️ 设置」→「📥 完整覆盖导入」→ 选文件
4. 如有未完成游戏,弹确认;确认后清空写入

> **iPad 上的游戏端没有「编辑」入口**——导入数据按钮可以放在游戏端的「设置」或开始页角落,不需要密码保护。

##### 图片处理策略

**MVP**:JSON 完全不含图片
- 题目无插图
- 拍的合影存在 iPad 本地,不通过 JSON 同步(用户自己长按存到相册)

**V1**:JSON 直接 base64 嵌入图片(题目插图、地图)
- 单个 L1 大小约 5-7MB,AirDrop/邮件/微信无压力
- **不引入 zip 包方案**(保持「JSON 是唯一中转」原则)
- 通关合影**仍然不进 JSON**,孩子拍完直接长按存 iPad 相册

#### 4.1.7 删除引用保护

**核心原则**:有引用关系的资源不允许直接删除,必须先在引用方解除引用。

**具体规则**:

**删除题目(L3)时**:
- 检查是否被任何 L2 引用(扫描 Point.questionIds)
- 如有引用 → 弹拒绝对话框,详见下方
- 如无引用 → 弹「确定删除『这道题』吗?」→ 确认即真正删除(IndexedDB 物理删除)

**删除 L2 点位时**:
- 直接弹「确定删除『凉亭』吗?」→ 确认即删
- 该 L2 引用的题目 `usedCount` **不减**(保持消耗式设计)

**删除 L1 探险时**:
- 弹「确定删除『周末小区探险』吗?这会同时删除其中所有点位」→ 确认
- 同时删除 L1 + 其包含的所有 L2
- L2 引用的题目 `usedCount` **不减**

**拒绝删除题目的对话框示例**:

```
不能删除「这道题」

这道题正在被以下点位使用:
- 周末小区探险 - 第 2 站凉亭
- 公园探险 - 第 1 站门口

请先去这些点位中移除引用,再来删除。

[确定]
```

V1 增强:增加「去查看引用」按钮,跳转到第一个引用的 L1 详情页。

**为什么不做 Undo toast**:

- 引用保护已经覆盖了"误删有价值数据"的场景
- 删除孤立题目的撤销价值低(可从 JSON 备份恢复)
- 删除关卡的撤销价值低(关卡可从 JSON 备份恢复)
- 简化代码 50%

#### 4.1.8 地图标记编辑器(V1)

**入口**:L1 详情页"📍 地图标记"按钮(已上传地图后才出现)。

**界面**(桌面端布局,**不支持 iPad 触摸编辑**):
- 左侧:地图图片显示区(按容器宽度自适应缩放)
- 右侧:操作面板(预览开关、清空所有按钮、L2 标记列表)

**交互**:
- **点击地图空白处放置标记**:自动按 L2 顺序绑定坐标(第 1 次点击 = 第 1 个 L2 的坐标,依此类推)
- **拖动已有标记**:调整位置(鼠标按住标记拖动到新位置)
- **点击已有标记**:弹小菜单(删除 / 取消)
- **预览开关**:
  - 关(默认编辑视图):显示工具标记(带数字 1/2/3/... 的圆圈,标识对应 L2 顺序)
  - 开(游戏端预览):显示游戏端最终效果(空心圆圈 + 名称,模拟"已发现未完成"状态)
- **清空所有按钮**:弹二次确认 → 删除所有坐标重标

**坐标存储**:百分比坐标(如 `x: 0.35, y: 0.62`),渲染时换算成像素

**保存限制**:
- 必须**所有 L2 都标了坐标**后,L1 才能保存
- 否则提示「还有 X 个点位没标坐标,无法保存」

**与点位的同步**:
- L2 名称变化 → 编辑器中预览模式下名称自动更新
- 删除某个 L2 → 该 L2 对应的地图坐标自动清空(提醒家长重新打开编辑器查看)
- 增加新 L2 → 新 L2 坐标为 null,需家长再次打开编辑器补标

> ⚠️ **游戏端使用独立的 IndexedDB**:数据库名 `treasure-hunt-play-db`,与编辑端的 `treasure-hunt-admin-db` 物理隔离。

#### 4.2.1 启动与首屏

**启动页(MVP 单人版)**:
- App 图标 + 「寻宝游戏」标题
- 大按钮「开始游戏」
- 小字「设置」(角落,含「导入数据」入口)

**探险选择页**:

- 点击卡片 → 直接进入「开始冒险」流程
- **MVP 不读 openingStory 字段**(即使数据里有也不显示),V1 启用故事背景页

#### 4.2.2 开始游戏(MVP 单人版)

- 显示探险名 + 大按钮「开始冒险!」

**V1 多人版会插入**:
- 人数选择 → 玩家创建 → 队名设置 → 开始

#### 4.2.3 游戏主流程(每个 L2 点位)

**1. 提示展示**
- 大字:当前是第几站
- 提示文本
- 大按钮「我找到了!扫码!」

**2. 扫码 / 输码**
- 全屏摄像头预览 + 扫码取景框
- 底部「输入数字码」按钮(扫不出时用)
- 底部「找不到?」按钮(二维码丢失/损坏时用,需家长输码)

**两个按钮的语义区分**(v1.7 澄清):

| 按钮 | 使用场景 | 最终行为 |
|------|---------|---------|
| **输入数字码** | 孩子找到了二维码但扫不出来(光线/手抖) | 输码后**进入答题**(跟扫码成功一样) |
| **找不到?** | 二维码物理损坏/丢失,孩子找不到 | 输码后**进入答题**(跟扫码成功一样) |

> 两个按钮**最终行为相同**(都进入答题),只是入口语义不同,方便孩子表达「问题在哪里」。家长可以问孩子「你是扫不出来还是找不到啊?」帮助判断现场情况。
> 
> 数字码从哪里来:家长手机/iPad 联网查编辑端、或翻应急小抄打印纸。如果两者都没有,这一局玩不下去,回家再玩(MVP 不做「真跳过」功能)。

**3. 扫码 / 短码结果**
- **当前点位**:「叮咚」+ “开始挑战”页 → 进入答题(MVP)
- **其他点位的码**:友好提示「这不是你现在要找的点哦!再看看提示~」
- **完全无效的码**:提示「这不是寻宝游戏的码哦」

**V1 增强**:扫码成功后插入「发现 → 地图浮现 → 引导」流程:

**3a. 发现 → 地图浮现 → 引导(V1 新增)**

扫码成功
↓
[若有发现文案] 弹窗显示发现文案 → 点击「继续」
↓
地图自动全屏弹出 + 该点位的圆圈和名称浮现动画(无印章,表示"已发现未完成")
↓ 动画结束 0.5 秒后
[若有答题引导] 弹窗显示答题引导 → 点击「开始答题」→ 关闭地图进入答题
[若无答题引导] 0.5 秒后自动关闭地图进入答题

**三段文案柔性逻辑**:
- 都填 → 走完整流程(发现弹窗 → 地图动画 → 引导弹窗 → 答题)
- 只填发现文案 → 发现弹窗 → 地图动画 → 自动进答题
- 只填答题引导 → 地图动画 → 引导弹窗 → 答题
- 都不填 → 地图动画 → 自动进答题
- 整个 L1 都没上传地图 → 跳过地图动画,扫码后直接按文案是否存在决定弹窗

**4. 多题连答**

一个 L2 点位有多道题,依次答:

- 显示「这一站有 3 道题,第 1 / 3 题」
- 每道题:题目 + 选项 + 求助按钮(位于**右上角**)
- 答题反馈机制同下面 §4.2.4
- 一道题答完进入下一道
- 所有题答完后,进入拍照环节

**5. 答题反馈(每道 L3 题)**
- **答对**:「叮咚」+ 绿色对勾 + 进入下一题
- **第 1 次答错**:温和提示「再想想?」可重选
- **第 2 次答错**:温和提示 + 显示题目提示(如有) + 可重选
- **第 3 次答错(保底)**:显示正确答案 + 「没关系,下次记住啦!」+ 自动通过(0 分)

**6. 答完所有题后的通关流程**

- 显示「📸 拍一张通关合影(可选)」按钮 + 「跳过」按钮

**拍合影**:
- 点拍照:调系统相机
- 拍完照片显示,下方提示「**长按图片可保存到系统相册**」
- 同时存到 IndexedDB(每 L2 一张)
- 照片压缩:1280px 长边,JPEG quality 0.7
- 拒绝相机权限或跳过:直接进入下一步

**核心**:每个 L2 点位**只在完成时拍一张**,不是每道题都拍。

**V1 增强**:答完题后插入「通关庆祝弹窗 → 地图盖章动画」:

[V1] 所有题答完
↓
[V1,若有通关庆祝文案] 弹窗显示通关庆祝文案 → 点击「继续」
↓
[V1] 地图自动全屏弹出 + 该点位盖章动画(红色五角星砸下 + 轻微震动 + 名字变金色,0.8 秒)
↓ 动画结束 0.5 秒后自动关闭地图
↓
[MVP & V1] 显示「📸 拍一张通关合影(可选)」按钮 + 「跳过」按钮

**V1 盖章时机说明**:
- 圆圈和名字的**浮现**发生在步骤 3a(扫码后)
- 印章的**盖下**发生在步骤 6(答完题后)
- 这两个动画在地图上是两次独立的状态变化,节奏清晰不抢戏

**7. 拍照后回扫码的摄像头处理**(v1.7 修订)

**问题**:从拍照流程返回扫码界面后,iPad Safari 的摄像头 API 可能不稳定,自动重启容易黑屏。

**MVP 方案(用按钮代替延迟)**:
- 拍完照后不自动进入扫码界面
- 显示一个大按钮「📷 继续扫码找下一站」
- 文案下方小字:「点击按钮重新打开摄像头」
- 孩子点击按钮 → 真正调用 `getUserMedia` 启动摄像头

**好处**:
- 不依赖 `setTimeout` 等待硬件就绪
- 孩子有「主动确认」的小仪式感
- 摄像头启动失败时,孩子知道再点一次

## 6. 数据模型

### 6.0 IndexedDB 数据库约定

**两端物理隔离 + 运行时拦截**(v1.7 强化):

| 端 | 数据库名 |
|---|---------|
| 编辑端 admin/ | `treasure-hunt-admin-db` |
| 游戏端 play/ | `treasure-hunt-play-db` |

**严禁**:跨端访问对方的数据库。所有数据流转必须通过 JSON 文件。

**运行时拦截**(v1.7 新增):`shared/db/db-config.js` 的初始化函数会检查当前页面路径与数据库名是否匹配:

```javascript
// shared/db/db-config.js 示例
export const ADMIN_DB_NAME = "treasure-hunt-admin-db";
export const PLAY_DB_NAME  = "treasure-hunt-play-db";

export function assertDbContext(dbName) {
  const isAdminPath = location.pathname.includes('/admin/');
  const isPlayPath = location.pathname.includes('/play/');
  
  if (isAdminPath && dbName !== ADMIN_DB_NAME) {
    throw new Error(`admin 路径不允许访问 ${dbName}`);
  }
  if (isPlayPath && dbName !== PLAY_DB_NAME) {
    throw new Error(`play 路径不允许访问 ${dbName}`);
  }
}
```

**ID 一致性约定**:
- JSON 文件里的所有 ID(L1 / L2 / 题目的短 ID)在导入时保持不变
- 编辑端创建数据时 ID 一旦生成永不修改
- 这是跨设备同步判重的基础
- MVP 单用户场景下 ID 碰撞概率几乎为零;V1 跨家长分享 L1 时需要重新评估

**短 ID 生成方式**(v1.7 明确):

```javascript
// shared/utils/id.js
function generateId(prefix) {
  const timestamp = Date.now().toString(36);      // 时间戳 base36
  const random = Math.random().toString(36).substr(2, 4);  // 4 位随机
  return `${prefix}_${timestamp}_${random}`;
}

// 用法:
generateId("lvl");  // → "lvl_lq2nx8j2_a3kf"
generateId("pt");   // → "pt_lq2nxa1f_b5kc"
generateId("q");    // → "q_lq2nxc2g_d8mn"
```

因为时间戳每毫秒变一次,**不需要写 IndexedDB 查重循环**。

#### 表结构

数据模型采用**独立表 + ID 引用**设计(不是嵌套)。Dexie.js 表定义:

**admin 端**(`treasure-hunt-admin-db`):
```javascript
db.version(1).stores({
  levels:    'id, name, updatedAt',
  points:    'id, levelId, order, code',
  questions: 'id, subject, difficulty, usedCount',
});
```

**play 端**(`treasure-hunt-play-db`):
```javascript
db.version(1).stores({
  levels:    'id, name',
  points:    'id, levelId, code',
  questions: 'id',
  sessions:  'id, levelId, status, startedAt',
  photos:    'id, type, ownerId',
});
```

> `'id, levelId, order'` 表示 `id` 是主键,`levelId` 和 `order` 是索引(查询快)。

### 6.7 JSON Envelope

JSON 导出文件的标准格式:

```javascript
{
  "schemaVersion": "1.0",
  "exportedAt": 1716800000000,
  "exportType": "full",      // MVP 只有 "full";V1 加 "single-level"
  "data": {
    "levels":    [Level, ...],
    "points":    [Point, ...],
    "questions": [Question, ...]
  }
}
```

**v1.7 修改**:
- 删除 `exportedBy` 字段(永远只有 "admin" 一个值,无任何代码使用)
- MVP 阶段 `schemaVersion` 固定 `"1.0"`,V1 启用地图功能时升 `"1.1"`(向后兼容)
- 1.1 在 Level 增加了 `mapImage / mapFontSize / mapNameColor / mapNameColorCompleted`
- 1.1 在 Point 增加了 `mapX / mapY / questionIntroText / completionText`
- 1.0 文件导入 1.1 App → 兼容,地图字段为 null(地图功能跳过)
- 1.1 文件导入 1.0 App → 主版本相同允许,但旧 App 忽略地图字段

**MVP 简化的版本检查逻辑**(v1.7):

MVP 只做最简检查,不解析主/次版本号:
```
1. 解析 JSON 文件
2. 检查 envelope 是否存在 schemaVersion 字段
   - 不存在 → 拒绝导入,提示「这不是有效的寻宝游戏数据文件」
3. 检查 schemaVersion === "1.0"
   - 不等 → 拒绝,提示「该文件由不兼容的 App 版本创建,请升级 App」
4. 通过 → 进入导入流程
```

V1 启用 1.1 后再加主/次版本解析逻辑。

**地图图片在 JSON 中的处理**:
- `mapImage` 字段为压缩后图片的 **base64 字符串**(约 200-400KB → base64 后约 250-500KB)
- 单个 L1 完整 JSON 大小约 300-600KB(无插图)~5-7MB(有插图),AirDrop / 邮件传输无压力
- **不引入 zip 包方案**(保持「JSON 是唯一中转」原则)

- **删除 L1**:弹「确定删除『周末小区探险』吗?这会同时删除其中所有点位」→ 确认即删
- **删除 L2**:弹「确定删除『凉亭』吗?」→ 确认即删(不影响题库本身)
- **删除题目时有引用**:弹拒绝对话框(详见 §4.1.7),列出引用关卡,提示用户先解除引用
- **删除题目无引用**:弹「确定删除『这道题』吗?」→ 确认即删
- **二维码数字码冲突**:生成时检查全表,冲突则重新生成
- **CSV 导入错误**:单行错误不影响其他行;显示成功/失败统计 + 失败原因
- **CSV 编码不对**(v1.7 修订):提示「请用 UTF-8 保存」+ Excel 截图教学。**不引入编码自动检测库**(jschardet 等)

**级联删除规则**:
由于数据模型是独立表设计,删除时需要手动级联:
- **删除 Level**:先删除所有 `levelId === 该 Level.id` 的 Points,再删除 Level 本身(不影响 Question 表)
- **删除 Point**:直接删除该 Point(不影响 Level,不影响 Question 的 usedCount)
- **删除 Question**:检查是否有任何 Point 的 `questionIds` 包含该题。有 → 拒绝;无 → 删除

### 7.2 游戏端

- **扫到其他 L2 的码**:友好提示「这不是现在要找的点哦」
- **扫到完全无效的码**:「这不是寻宝游戏的码哦」
- **二维码物理损坏**:「找不到?」按钮 → 输码 → 进入答题
- **扫码不清晰**:「输入数字码」按钮 → 输 6 位数字 → 进入答题
- **答题中切换 App**:恢复后回到答题状态(自动保存机制)
- **拍照权限拒绝**:不阻塞,直接进下一步
- **iOS Safari 切后台黑屏**:监听 visibilitychange,切回时重新初始化摄像头
- **扫码后立即关闭摄像头流**,需重启时显示「📷 继续扫码」按钮让用户主动触发(v1.7 统一,详见 §4.2.3)
- **拍照后回扫码**(v1.7 修订):显示「📷 继续扫码找下一站」按钮,孩子点击后才启动摄像头(不再使用 setTimeout 等待 1.5 秒)
- **设备旋转**:游戏中竖屏 ↔ 横屏自动适配但不强制
- **电量过低**:提示「iPad 没电啦,建议先充电」
- **网络环境**:游戏端 PWA 调用摄像头需要安全上下文(HTTPS 或 localhost)。本地测试时**必须用 localhost**(不能用局域网 IP 如 192.168.x.x),或部署到 GitHub Pages 用 HTTPS 测试

### 7.3 数据安全

**iOS 清理 PWA 数据风险**:
- 提示用户定期开启游戏端(保持「活跃」状态)
- 重要数据在编辑端导出 JSON 备份
- 拍的照片提示「长按保存到相册」

**iOS 用户可手动清除 PWA 数据**:
- iOS 16+ 在「设置 → Safari → 高级 → 网站数据」里可手动删除 PWA 数据
- 操作很简单,小孩可能无意触发
- 建议家长**不要让孩子随便点 iPad 系统设置**
- 一旦数据被清,游戏进度无法找回;但只要电脑端 JSON 备份还在,关卡和题库可以从 JSON 恢复

**编辑端浏览器数据丢失**:
- 用户自己保管 JSON 备份
- 多设备共用:每次导出 / 导入 同一份 JSON
- 提示用户:「JSON 文件就是你的数据库,请妥善保管」

### 7.4 数据导入冲突

**MVP 整库覆盖导入规则**:
- 文件类型必须是 `full`,其他类型拒绝并提示「这不是整库备份文件」
- 缺少 schemaVersion 字段:拒绝并提示「这不是有效的寻宝游戏数据文件」
- schemaVersion 不等于 "1.0":拒绝并提示「该文件由不兼容的 App 版本创建,请升级 App」
- 通过检查后:
  - 编辑端:弹「这将替换当前所有数据,确定吗?」→ 确认后清空写入
  - 游戏端:先检查未完成游戏 → 有则弹「检测到正在进行中的游戏,导入会清除进度,确定吗?」→ 再弹「这将替换当前所有数据,确定吗?」→ 确认后清空写入

**V1 追加单 L1 导入规则**(v1.7 推迟到 V1):
- 文件类型必须是 `single-level`,其他类型拒绝并提示「这个功能只接受单个探险的 JSON 文件」
- L1 ID 不冲突 → 直接新增,不弹窗
- L1 ID 冲突 → 弹覆盖确认弹窗(详见 §4.1.6)
- 题目 ID 判重 + 内容冲突显示「N 道题将被更新内容」
- **usedCount 取「iPad 现有值」和「JSON 值」的较大值**
- 待替换的 L1 上正好有未完成游戏:弹「这个探险正在玩中,替换会清除进度,确定吗?」(仅游戏端)

### 7.5 JSON Schema 版本检查(v1.7 简化)

**MVP 阶段不解析主/次版本号**,只做最简检查:

```
1. 解析 JSON 文件
2. 检查 envelope 是否存在 schemaVersion 字段
   - 不存在 → 拒绝
3. 检查 schemaVersion === "1.0"
   - 不等 → 拒绝
4. 通过 → 进入导入流程
```

**当前 schemaVersion 历史**:
- 1.0(MVP 起始版本)
- 1.1(V1 启用地图字段时升级,向后兼容)

**何时升级主版本号**:数据结构发生不兼容变更(字段含义变化、必填字段新增等)

**何时升级次版本号**:新增可选字段(向后兼容)

V1 启用 1.1 后,导入逻辑需要扩展为「主版本相同允许,次版本不同警告」。详细规则到 V1 设计时再写。

### 7.6 Service Worker 更新(游戏端)

**v1.7 修订:MVP 阶段直接采用动态缓存策略**:

```javascript
// service-worker.js 简化伪代码
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.open('treasure-hunt-runtime').then(cache =>
      cache.match(event.request).then(cached => {
        const networkFetch = fetch(event.request).then(response => {
          // 把成功的响应缓存起来(包括 CDN 资源)
          if (response.ok) cache.put(event.request, response.clone());
          return response;
        }).catch(() => cached);  // 网络失败则用缓存
        return cached || networkFetch;
      })
    )
  );
});
```

**好处**:
- 不需要手写维护 `FILES_TO_CACHE` 列表
- CDN 资源(Dexie、html5-qrcode、qrcode.js、PapaParse)自动被缓存,离线可用
- 新增/重命名 `play/` 下的静态文件不需要同步改 SW

**注意**:
- 第一次访问游戏端**必须有网**(让 SW 把所有资源缓存住)
- 启动时后台检查新版本
- 有新版本顶部显示「点此刷新」按钮
